How I Built a Cybersecurity Portfolio Through Open Source Contributions

A practical guide to building visibility and credibility in the security community through meaningful open source work.

The Problem

Everyone says "contribute to open source" but nobody explains how to do it effectively when you're starting out. After spending weeks contributing to major cybersecurity projects, I learned that it's not just about writing code — it's about being helpful.

The Strategy

1. Start with Documentation

Documentation PRs are the easiest entry point. I found a broken link in IntelOwl's OpenCTI documentation:

# Before
url = "https://docs.opencti.io/latest/deployment/..."

# After  
url = "https://docs.opencti.io/latest/deployment/authentication"

Why it works: Maintainers love documentation fixes because they improve user experience without risk.

2. Fix Real Bugs

I found an issue in Prowler where stopped EC2 instances weren't being checked after a specific number of days. The check existed but wasn't enabled by default.

What I did:

  • Created a new check with proper severity
  • Added comprehensive docstrings
  • Fixed timezone handling (datetime.now(timezone.utc) instead of naive datetime)
  • Addressed CodeRabbit AI feedback professionally

3. Extend Existing Rules

For Checkov, I extended CKV_GIT_3 to also check for GitHub repository vulnerability alerts. This is valuable because:

  • It improves security coverage
  • It's backward compatible
  • It follows existing patterns

4. Comment Helpfully on Issues

This is the most underrated strategy. I didn't just say "+1" or "I have this too." Instead:

## Root Cause Analysis

The issue occurs because Elasticsearch uses base64url IDs 
(44 characters, [A-Za-z0-9_-]) while Kubernetes labels 
require [a-z0-9] with optional [-.] separators.

## Suggested Fix

import base64

def sanitize_for_k8s(es_id: str) -> str:
    decoded = base64.urlsafe_b64decode(es_id + "==")
    return ''.join(c if c.isalnum() or c in '-.' else '-' for c in decoded)[:63]

This approach:
1. Decodes the Elasticsearch ID
2. Sanitizes invalid characters
3. Respects the 63-character label limit

Why it works: You're not just identifying problems — you're solving them.

5. Build Tools That Solve Real Problems

I created three tools that fill gaps in the ecosystem:

Tool Purpose Why It's Useful
ioc-extractor Extract IOCs from logs Supports 17 IOC types including JA3, AWS keys, GitHub tokens
sigma-tester Validate Sigma rules Catches issues before deployment
yara-tester Validate YARA rules Performance warnings and quality checks

Key insight: These tools are pure Python with no external dependencies (except optional yara-python). This makes them easy to adopt.

The Impact

What I Learned

  1. Quality > Quantity: One helpful comment beats 10 "+1" comments
  2. Be Specific: Reference exact files, line numbers, and functions
  3. Follow Up: Respond to feedback professionally
  4. Think Like a Maintainer: What would help them most?

The Framework

Here's my repeatable process:

1. FIND: Look for issues with labels like "question", "bug", "help wanted"
2. ANALYZE: Read the issue thoroughly, understand the root cause
3. SOLVE: Provide a specific solution with code examples
4. DOCUMENT: Create tools or write guides that help others
5. FOLLOW UP: Respond to feedback and iterate

Conclusion

Open source contribution isn't just about code. It's about being a helpful member of the community. By combining documentation fixes, bug fixes, rule extensions, helpful comments, and tool creation, you build a portfolio that demonstrates real-world skills.

The best part? This approach works for anyone, regardless of experience level. You don't need to be a senior engineer to fix a broken link or answer a question helpfully.

Start small, be consistent, and always focus on being helpful.
